# packet

`packet` is a standalone external C++ utility intended to become a **general GW2 reverse-engineering helper**.

Its design goal is to keep heavy analysis, correlation, and experiment-specific tooling
**outside** the injected DLL, so the in-process module can stay small and focused while
this external tool grows into the main research workspace.

## Purpose

`packet` is meant to be the general-purpose companion tool for:

- packet/candidate message research
- cooldown state reconstruction
- memory anchor extraction from logs
- x64dbg workflow support
- future `ReadProcessMemory`-based external inspection
- report generation and experiment comparison

In other words:

- the injected DLL should become a lightweight probe
- `packet` should become the main external analysis tool

## Current scope

The current implementation is intentionally narrow and stable.

Right now it can:

- parse `cd-log` files generated by the in-process DLL
- reconstruct cooldown runs from `[CDHUNT]`
- collect key memory anchors from `[SKMEM]`
- print x64dbg-ready address variables for a chosen run

This gives us a clean first step: extract run structure and debugger inputs without
adding more complexity to the injected project.

## Build

Recommended on this machine:

```powershell
cmd /c .\build-packet.cmd
```

This uses the local MSVC toolchain directly and writes the executable to:

```text
build\Release\packet.exe
```

If you later want to switch back to a full CMake workflow, the `CMakeLists.txt`
is already present.

## Commands

### Summary

```powershell
packet summary --log "C:\path\to\20260420.log"
```

Prints all parsed runs with:

- `run_id`
- hotkey
- `SKILL_PRESS`
- `NODE_CD_START`
- `NODE_READY`
- count of `SKMEM` phases seen

### Latest

```powershell
packet latest --log-root "C:\Users\Charlie\Desktop\gw2\kx-packet-inspector-main\x64\Release\cd-log"
```

Finds the newest `.log` in the directory and prints the same summary.

### x64dbg

```powershell
packet x64 --log "C:\path\to\20260420.log" --run 1
```

Prints the best-known addresses for that run:

- `ctx`
- `chcli_ctx`
- `local`
- `chcli_skillbar`
- `char_skillbar`
- `ptr40_value`
- `ptr48_value`
- `ptr60_value`
- `ptr40_slot`
- `ptr48_slot`
- `ptr60_slot`
- `focus_slot_addr`

## Current limitations

This first version does **not** yet:

- parse full packet timelines
- rebuild `0x0017` / `0x0028` stage tables
- parse `[DBGASST]`, `[NODEVAL]`, `[PAYDIFF]`
- reconstruct `PTR40` / `PTR60` transition graphs
- generate x64dbg scripts directly
- attach to the game process
- read process memory directly
- scan packets or memory live
- export markdown / CSV reports

So today it is best understood as:

- a run/log parser
- an address extraction helper
- a bridge from `cd-log` to x64dbg

## Planned features

The intended roadmap is to expand `packet` into a more general tool with modules like:

### 1. Log analysis

- parse `[CDHUNT]`, `[SKMEM]`, `[DBGASST]`, `[NODEVAL]`, `[PAYDIFF]`
- extract run timelines
- correlate packet and memory events
- compare multiple runs across one or more log files

### 2. Cooldown state analysis

- rebuild `PTR40 -> enter_cd` candidate transitions
- rebuild `PTR60 -> leave_cd` candidate transitions
- compare cooldown state chains across skills
- track state rebuilds under alacrity / charge / modifier scenarios

### 3. Packet research

- summarize candidate headers and subtypes
- compare `0x0001`, `0x0015`, `0x0017`, `0x0028` across runs
- detect stable packet shapes near memory state transitions
- help test “success packet / opcode reuse” hypotheses

### 4. Debugger integration

- generate x64dbg-ready variable blocks
- generate x64dbg scripts directly for chosen runs
- generate watch lists for `char_skillbar`, `ptr40_slot`, `ptr48_slot`, `ptr60_slot`, `focus_slot_addr`

### 5. External memory tooling

- optional `OpenProcess` / `ReadProcessMemory` helpers
- live snapshot comparison outside the injected DLL
- pointer-chain dumps using addresses learned from logs

### 6. Report/export

- markdown summaries
- CSV exports
- run comparison tables
- checklist-style validation reports

## Long-term direction

Long term, `packet` should be treated as a **general external tooling project**, not a
single-purpose cooldown script.

Cooldown research is just the first use case.

The intended end state is:

- lightweight in-game probe inside the GW2 process
- reusable external analysis tool outside the process
- debugger-friendly outputs that shorten reverse-engineering cycles

## Near-term next steps

- parse `[DBGASST]`, `[NODEVAL]`, `[PAYDIFF]`
- correlate `PTR40 / PTR60` state transitions against packets
- generate x64dbg scripts directly
- add optional live `ReadProcessMemory` helpers
- export markdown / CSV reports
