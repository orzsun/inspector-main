# x64dbg 操作整理

这份文档整理的是当前已经跑通过、并且和项目日志能对上的 x64dbg 使用流程。

目标不是在文档里下结论说某个 packet 或某个状态就是 `cooldown_end`，而是把目前可复现的调试动作、观察点、以及和项目日志的对应关系记清楚。

## 当前调试目标

当前主线是：

- 用项目日志里的 `DBGASST / ZNODE / ZSWITCH / ZCLTABLE` 作为时间轴和状态簇标记
- 在 x64dbg 里观察与这些状态簇切换相关的代码路径
- 区分：
  - 节点对象内部字段更新
  - 更上层的状态簇切换路径

当前不做这些事：

- 不把 `0x0017` 或 `0x0028` 直接命名成 `cooldown_end`
- 不把 `NODE_READY_BASELINE` 当成真正 ready 回归锚点
- 不把所有非 ready 状态继续压成 `UNKNOWN / NODE_OTHER`

## 每次重开游戏后的第一步

游戏重开后，运行时地址通常会变，所以每次都要先从**最新日志**里拿一条 `Z` 的 `DBGASST t0`。

需要提取的字段：

- `ptr48_untagged`
- `ptr60`
- `inferred_base`

示例：

```text
[DBGASST] phase=t0 hotkey=Z ... ptr48_untagged=0x1ee430d7490 ptr60=0x1ee430d74b0 inferred_base=0x1ee430d7450 ...
```

用它计算这轮的槽位地址：

- `slot40 = inferred_base + 0x40`
- `slot60 = inferred_base + 0x60`
- `ptr48+0x20 = ptr48_untagged + 0x20`

当前最新一轮样本里：

- `slot40 = 0x1EE430D7490`
- `slot60 = 0x1EE430D74B0`
- `ptr48+0x20 = 0x1EE430D74B0`

说明在这轮里：

- `slot60`
- `ptr48+0x20`

是同一个地址。

## x64dbg 运行脚本的前提

x64dbg 的脚本和命令需要在**调试器暂停状态**下执行。

标准顺序：

1. 附加到 `Gw2-64.exe`
2. 手动暂停
3. 执行脚本或命令
4. 再 `F9` 继续运行

如果看到：

```text
显示调试器只有在暂停时才能运行脚本
```

说明只是执行时机不对，不是脚本内容本身有问题。

## 已验证过的两类断点

### 1. 槽位/对象写断点

对最新一轮的地址下硬件写断点，例如：

- `0x1EE430D7490`
- `0x1EE430D74B0`

设置：

- `Hardware breakpoint`
- `Write`
- `8 bytes`

这类断点已经证实会命中到和 Z 状态对象相关的代码路径。

### 2. 代码地址日志断点

已经实际命中过、并且值得继续观察的代码地址包括：

- `gw2-64.00007FF62A90A4F8`
- `gw2-64.00007FF62B8C8412`
- `gw2-64.00007FF62B8C84D2`

其中：

- `0x7FF62A90A4F8` 已被多次命中
- `0x7FF62B8C8412 / 0x7FF62B8C84D2` 是从后续日志中浮出来的上层处理候选点

## 已经确认的一个重要现象

对 `slot40 / slot60` 下写断点时，多次命中到：

```asm
mov qword ptr ds:[r14+0x08], rsi
```

对应地址：

- `gw2-64.00007FF62A90A4F8`

这说明：

- 命中的不是主槽位 `+0x00`
- 而是对象内部的 `+0x08` 字段写入

所以当前可以先把这个地址理解为：

- **节点对象内部字段更新点**

而不是最终的“状态簇切换总入口”。

- 目前可以视作“与 Z 节点状态更新有关的真实代码路径”

## 当前最值得继续追的代码点

基于已有日志，当前推荐优先继续观察的是：

- `0x7FF62A90A4F8`
- `0x7FF62B8C8412`
- `0x7FF62B8C84D2`

其中：

- `0x7FF62A90A4F8` 已多次命中，确认是对象内部字段更新点
- `0x7FF62B8C8412 / 0x7FF62B8C84D2` 与后续状态对象处理更接近，值得继续做日志断点

## 已记录过的关键寄存器特征

`0x7FF62A90A4F8` 命中时常见特征：

- 指令：`mov qword ptr ds:[r14+0x08], rsi`
- `r14` 指向目标对象基址
- `rsi` 是要写入到该对象 `+0x08` 的指针值

所以断下来时最值得优先看的是：

- `R14`
- `RSI`
- `RCX`
- `RDX`
- `R8`
- `R9`

其中：

- `R14` 用来看当前写入目标对象是谁
- `RSI` 用来看被挂接进去的新指针是谁
- `RCX/RDX/R8/R9` 用来看更上层调用参数像不像容器对象或状态机上下文

## 当前脚本记录的重点

当前脚本主要输出这些标签：

- `[ZHW40]`
- `[ZHW60]`
- `[ZCODE_A4F8]`
- `[ZCODE_8412]`
- 以及对应的 `STACK_BEGIN / STACK_END`

这些日志用来回答两类问题：

1. 是不是在命中正确的对象链路
2. 当前命中的代码点更像：
   - 节点对象内部更新
   - 还是上层状态簇处理入口

## 如何判断“这次断点有价值”

有价值的断点现场通常满足这些条件：

- `RIP` 在 `gw2-64.exe` 模块内部
- 不是 `ntdll / kernelbase` 等等待线程现场
- 调用栈前几层也主要是 `gw2-64`
- 写入地址或关键寄存器能和：
  - `slot40`
  - `slot60`
  - 已知状态簇节点地址
  对上

如果断下来看到的是：

- `WaitForSingleObjectEx`
- `ntdll`
- 调用栈里主要是系统 DLL

那基本就是噪音，不必继续分析。

## 如何和项目日志对照

调试时要把 x64dbg 观察和项目日志一起看。

重点对照这些标签：

- `[DBGASST]`
- `[ZNODE]`
- `[ZSWITCH]`
- `[ZCLTABLE]`
- `[CD0017G]`
- `[CD0028]`

当前原则：

- `ZNODE / ZSWITCH / ZCLTABLE` 用来标记**状态簇**
- `0x0017 / 0x0028` 只做**时间轴标记**
- 不把 `0x0017 / 0x0028` 直接命名成 ready 或 cooldown_end

## 当前已知限制

1. 节点对象地址会随游戏重开而变化，所以每轮都必须刷新 `DBGASST t0`
2. 命中 `0x7FF62A90A4F8` 只能说明到了对象更新链，不等于已经找到最上层切换入口
3. `NODE_READY_BASELINE` 不能再用于 packet 的 ready 锚点统计
4. packet 时序统计只有在真正出现 `NODE_READY` 回归时才可用于 near-ready 分析

## 当前结论

目前 x64dbg 这边已经确认：

- 已经命中与 Z 状态对象有关的真实代码路径
- `0x7FF62A90A4F8` 是节点内部字段更新点
- 更值得继续观察的上层候选点是：
  - `0x7FF62B8C8412`
  - `0x7FF62B8C84D2`

当前还没有确认：

- 哪个函数是最终的状态簇切换总入口
- 哪些簇等价于“可释放 / 转好 / 后段 idle”

所以当前阶段最合理的做法是：

- 继续保留日志脚本
- 继续按最新 `DBGASST t0` 刷新地址
- 把 x64dbg 命中点当成“状态链路观察口”
- 不过早给状态簇下高层语义结论
